# Agent 路由与通信规则 v2.0

> **目的**: 定义消息路由和 Agent 间协作规范
> **原则**: 按职责路由，通过 Jerry 协调，不直接互相调用

---

## 一、路由策略

### 1.1 默认路由

| Channel | 默认 Agent | 说明 |
|---------|-----------|------|
| 企业微信 (wecom) | Jerry 🦞 | 所有消息先到 Jerry，由 Jerry 分发 |
| 微信小程序 (openclawwechat) | Jerry 🦞 | 所有消息先到 Jerry |
| WebChat | Jerry 🦞 | Rex 直接对话 |

### 1.2 关键词路由

| 关键词 | 路由到 | 说明 |
|--------|--------|------|
| 合同、审批、OA、法务 | Ella 🦊 | 合同相关 |
| 项目、ONES、进度、里程碑 | Oliver 🐘 | 项目相关 |
| 经营、报表、预算、KPI | Aaron 🦉 | 经营相关 |
| 邮件、通知、日程、文档 | Iris 🐦‍⬛ | 辅助相关 |
| 技术瓶颈、API 问题 | Iris 🐦‍⬛ | 技术瓶颈研究 |
| 其他 | Jerry 🦞 | 默认协调 |

### 1.3 命令路由

| 命令 | 路由到 | 说明 |
|------|--------|------|
| `/ella` | Ella 🦊 | 直接对话 Ella |
| `/oliver` | Oliver 🐘 | 直接对话 Oliver |
| `/aaron` | Aaron 🦉 | 直接对话 Aaron |
| `/iris` | Iris 🐦‍⬛ | 直接对话 Iris |
| `/jerry` | Jerry 🦞 | 直接对话 Jerry |

## 二、Jerry 分发机制

```
用户消息 → Jerry 分析意图
   │
   ├── 合同/OA → 转发 Ella
   ├── 项目/ONES → 转发 Oliver
   ├── 经营/报表 → 转发 Aaron
   ├── 邮件/辅助 → 转发 Iris
   ├── 技术瓶颈 → 转交 Iris (Agent Loop 研究)
   └── 其他/协调 → Jerry 处理
```

---

## 三、Agent 间通信

### 3.1 通信架构

```
┌─────────────────────────────────────────┐
│                Jerry 🦞                  │
│           (任务协调中心)                  │
│                                         │
│  接收 Rex 指令 → 分析意图 → 分发任务     │
│  收集 Agent 结果 → 汇总 → 汇报 Rex       │
└─────────────┬───────────────────────────┘
              │
     ┌────────┼────────┬────────┐
     │        │        │        │
  Ella 🦊  Oliver 🐘 Aaron 🦉 Iris 🐦‍⬛
```

### 3.2 任务分发

Jerry 通过 `sessions_spawn` 向 Agent 分发任务：

```python
sessions_spawn(
    agentId="ella",
    task="请审核这份合同：{合同内容}",
    mode="run"
)
```

### 3.3 结果返回

```
Agent 执行任务 → 生成结果报告 → 返回给 Jerry → Jerry 汇总 → 汇报 Rex
```

### 3.4 技术瓶颈处理流程

```
Agent 遇到技术瓶颈
   │
   ▼
1. 记录问题详情 (memory/YYYY-MM-DD.md)
   │
   ▼
2. 尝试自动解决 (最多 3 次)
   │
   ▼ 失败
3. 上报 Jerry (包含问题详情和已尝试方案)
   │
   ▼
4. Jerry 分析 → 转交 Iris (Agent Loop)
   │
   ▼
5. Iris 研究解决方案
   ├─ 搜索 (tavily/websearch/ClawHub/GitHub)
   ├─ 测试 (创建测试脚本)
   └─ 记录 (研究过程和结果)
   │
   ▼
6. Iris 返回方案 → Jerry 转发给原 Agent
   │
   ▼
7. 原 Agent 执行方案 → 完成任务
```

## 四、协作场景

| 场景 | 流程 | 说明 |
|------|------|------|
| 合同 + 项目 | Ella → Jerry → Oliver | 合同通过后创建项目 |
| 项目 + 经营 | Oliver → Jerry → Aaron | 项目进度同步用于经营分析 |
| 邮件 + 合同 | Iris → Jerry → Ella | 合同邮件转发审批 |
| 技术瓶颈 | 任何 Agent → Jerry → Iris | Iris 研究后移交方案 |
| 自进化闭环 | Iris 研究 → 记录 → 优化 → 移交 | Iris 持续优化研究方法 |

## 五、通信协议

### 5.1 任务格式

```markdown
# 任务: {任务名称}

## 来源
- 发起 Agent: Jerry
- 原始请求: {Rex 的原始指令}

## 要求
- {具体要求 1}
- {具体要求 2}

## 截止时间
- {P0: 立即 / P1: 24h / P2: 常规}
```

### 5.2 结果格式

```markdown
# 结果: {任务名称}

## 执行状态
- ✅ 完成 / ⚠️ 部分完成 / ❌ 失败

## 结果内容
- {具体内容}

## 异常信息
- {如有异常，记录详情}
```

### 5.3 技术瓶颈移交格式

```markdown
# 技术瓶颈移交

## 问题描述
- **来源 Agent**: {name}
- **问题**: {详细描述}
- **已尝试方案**:
  1. {方案 1} - 结果: {成功/失败}
  2. {方案 2} - 结果: {成功/失败}

## 研究要求
- **期望输出**: {期望的解决方案}
- **测试要求**: {需要的测试验证}
- **优先级**: P0/P1/P2
```

## 六、注意事项

1. **不直接调用**: Agent 间不直接调用，必须通过 Jerry 协调
2. **记录所有通信**: 每次任务分发和结果都记录到 memory
3. **超时处理**: 任务超过预期时间未完成，Jerry 主动跟进
4. **权限隔离**: 每个 Agent 只能访问自己的工作区
5. **技术瓶颈**: 统一由 Iris 研究解决，完成后移交方案
6. **自进化**: Iris 研究结果沉淀到知识库，供团队复用
